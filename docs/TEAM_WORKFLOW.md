# Руководство по командной разработке (Team Workflow)

Этот документ описывает процессы совместной работы над проектом TmWork, чтобы минимизировать конфликты в коде и обеспечить высокое качество продукта.

## 1. Основной процесс (Git Flow)

Мы используем упрощенную модель **GitHub Flow**.

1.  **Ветка `main`**: Содержит стабильный код. Прямые коммиты в эту ветку **строго запрещены**.
2.  **Создание ветки**: Для выполнения любой задачи создайте новую ветку от `main`:
    ```bash
    git checkout -b feature/название-задачи
    ```
3.  **Разработка**: Делайте небольшие, логически завершенные коммиты.
4.  **Синхронизация**: Регулярно подтягивайте свежие изменения из `main` в свою ветку, чтобы избежать сложных конфликтов в конце:
    ```bash
    git fetch origin
    git merge origin/main
    ```
5.  **Pull Request (PR)**: Когда задача готова, отправьте ветку на сервер и откройте PR в GitHub.

## 2. Code Review (Обзор кода)

Ни одно изменение не попадает в `main` без подтверждения (Approve) хотя бы от одного другого разработчика.

**На что смотреть при ревью:**
- Соответствует ли код архитектуре (Services, Decoupled logic)?
- Нет ли "магических чисел"?
- Покрыты ли изменения тестами?
- Не ломает ли этот код существующие функции?

## 3. Управление задачами

Для координации работы используем **GitHub Issues** и **Project Board** (Kanban-доску).

- **To Do**: Задачи, которые нужно сделать.
- **In Progress**: Задачи, над которыми сейчас работают. У каждой задачи есть один ответственный (Assignee).
- **Review**: Создан PR, ожидается проверка.
- **Done**: Задача в `main`.

> [!IMPORTANT]
> Один разработчик = Одно приложение (app) или Один компонент в один момент времени. Старайтесь не менять файлы, над которыми сейчас работает ваш коллега.

## 4. Конфликты и их решение

Если два человека изменили один и тот же файл, Git сообщит о конфликте при слиянии.

1.  **Связь**: Напишите коллеге, если видите, что его изменения пересекаются с вашими.
2.  **Архитектура**: Если конфликты случаются слишком часто, значит, логика слишком сильно связана. Используйте `services.py`, чтобы разделить "что делать" от "как это хранится".

## 5. Автоматизация

Перед пушем кода всегда запускайте:
- Линтер: `npm run lint` (фронтенд) и `flake8` (бэкенд).
- Тесты: `pytest` и `npx playwright test`.

Если тесты не проходят — PR не будет принят.
