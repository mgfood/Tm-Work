# TmWork Backend - Техническое руководство разработчика

Данный документ предназначен для разработчиков, которые будут поддерживать и расширять бэкенд платформы TmWork. Проект разработан с упором на масштабируемость, безопасность и чистоту архитектуры.

---

## 🏗 Архитектура

Проект следует принципам **API-first** и **Decoupled Architecture**. Бэкенд является чистым REST API и полностью независим от фронтенда.

### Основные принципы:
1.  **Thin Views (Тонкие контроллеры)**: Вьюхи (Views) отвечают только за обработку запросов, проверку прав доступа и возврат ответов. Они не содержат бизнес-логику.
2.  **Service Layer (Слой сервисов)**: Вся сложная бизнес-логика (расчеты, переходы статусов, работа с деньгами) находится в файлах `services.py` каждого приложения.
3.  **Immutable Transactions (Неизменяемые транзакции)**: Любое движение средств фиксируется в модели `Transaction`. Транзакции нельзя изменять или удалять.
4.  **Deterministic Lifecycle**: Жизненный цикл заказов (Jobs) и финансов (Escrow) четко определен через наборы статусов и валидацию переходов.

## 🛠 Технологический стек

- **Python 3.11+**
- **Django 5.2+**
- **Django REST Framework (DRF)**
- **JWT (SimpleJWT)** — для аутентификации.
- **drf-spectacular** — для автоматической генерации Swagger документации.
- **PostgreSQL** — основная БД для продакшена.
- **SQLite3** — доступна для локальной разработки через переключатель в `.env`.

## 📂 Структура приложений (Apps)

Проект разбит на приложения по зонам ответственности:

### 1. `users`
Управляет пользователями и правами доступа.
-   **Модель `User`**: Использует Email вместо username.
-   **Модели `Role` и `Permission`**: Гибкая ролевая модель (КЛИЕНТ, ФРИЛАНСЕР). Права доступа зависят от кодовых имен разрешений.
-   **Аутентификация**: JWT (Access + Refresh токены).

### 2. `profiles`
Публичные и приватные данные пользователей.
-   **Модель `Profile`**: Содержит био, аватар, контакты и рейтинги.
-   **Модель `Skill`**: Список навыков (Tags).
-   **Рейтинги**: Раздельные рейтинги для ролей Клиента и Фрилансера.

### 3. `jobs` (Заказы)
Основная сущность платформы.
-   **Жизненный цикл (Status)**: `DRAFT` -> `PUBLISHED` -> `IN_PROGRESS` -> `SUBMITTED` -> `COMPLETED` / `DISPUTE` / `CANCELLED`.
-   **Сервис `JobService`**: Управляет переходами статусов и бизнес-правилами (например, только клиент может отменить заказ до начала работы).

### 4. `proposals` (Отклики)
Связующее звено между фрилансером и заказом.
-   **Логика**: Один фрилансер может оставить только один отклик на один заказ.
-   **Принятие**: Принятие отклика автоматически переводит заказ в `IN_PROGRESS` и инициирует создание записи в `Escrow`.

### 5. `escrow` (Безопасность сделок)
Управление деньгами в процессе выполнения заказа.
-   **Статусы**: `CREATED` -> `FUNDS_LOCKED` -> `RELEASED` / `REFUNDED`.
-   **Безопасность**: Деньги блокируются при старте и выплачиваются только после подтверждения выполнения.

### 6. `transactions` (Аудит)
Журнал всех финансовых операций.
-   **Типы**: DEPOSIT, WITHDRAWAL, ESCROW_LOCK, ESCROW_RELEASE, FEE и др.
-   **Сервис `TransactionService`**: Единственный способ создания транзакций.

## 🔐 Безопасность

- **Никогда не доверяйте входным данным фронтенда**. Балансы и статусы рассчитываются только на бэкенде.
- **Custom Permissions**: Используйте `IsClient`, `IsFreelancer`, `IsJobOwner` для защиты эндпоинтов.
- **JWT**: Access-токен короткоживущий (60 мин), Refresh-токен требует ротации.

## 🚀 Настройка и запуск

### 1. Окружение
Скопируйте `.env.example` в `.env` и настройте переменные.
Ключевая настройка для локальной разработки:
```bash
DB_ENGINE=sqlite3  # Чтобы не устанавливать PostgreSQL на Windows
```

### 2. Установка
```bash
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r backend/requirements/base.txt
```

### 3. База данных
```bash
python backend/manage.py makemigrations
python backend/manage.py migrate
python backend/manage.py createsuperuser
```

### 4. Документация (Swagger)
Запустите сервер и перейдите по адресу:
[http://127.0.0.1:8000/api/schema/swagger-ui/](http://127.0.0.1:8000/api/schema/swagger-ui/)
Здесь можно тестировать API без Postman.

## 🔧 Как добавлять новые функции

1.  **Новая модель**: Создайте модель в соответствующем приложении.
2.  **Миграции**: `python manage.py makemigrations <app_name>`.
3.  **Логика**: Если действие сложное, создайте метод в `services.py`.
4.  **Serializer**: Создайте сериализатор в `serializers.py`.
5.  **View**: Создайте ViewSet в `views.py`, наследуясь от `ModelViewSet` или `GenericAPIView`.
6.  **URL**: Зарегистрируйте эндпоинт в `urls.py` приложения и в `api/v1/urls.py`.

---

## 📝 Заметки по поддержке

- **Валидация**: Используйте метод `validate()` в сериализаторах для бизнес-валидации (например, проверка даты дедлайна).
- **Ошибки**: Возвращайте ошибки через `serializers.ValidationError` для консистентных ответов 400 Bad Request.
- **Логирование**: Рекомендуется добавить логирование для критических операций в `services.py`.
