# ТЗ: Создание системы автотестирования TmWork

## 1. Цели
- Гарантировать стабильность критических путей пользователя.
- Уменьшить количество регрессионных ошибок при обновлениях.
- Автоматизировать проверку финансовой логики (Escrow, Транзакции).

## 2. Стек технологий
- **Backend (Python/Django)**: `Pytest` + `Django Test Framework`.
- **API**: `DRF APIClient`.
- **Frontend (E2E)**: `Playwright` или `Cypress`.
- **CI/CD**: GitHub Actions.

## 3. Области покрытия

### 3.1. Бэкенд (Unit & Integration)
- **Аутентификация**: Регистрация, логин, JWT (access/refresh), сброс пароля.
- **Пользователи**: Смена ролей (Client/Freelancer), права доступа (Permissions).
- **Заказы (Jobs)**: Жизненный цикл заказа (Draft -> Published -> ... -> Completed). Валидация переходов статусов.
- **Escrow**: Блокировка средств при принятии отклика, выплата после завершения, возврат средств.
- **Транзакции**: Создание записей о транзакциях при каждом движении денег. Неизменяемость записей.

### 3.2. Фронтенд (E2E тесты)
- **Critical Path 1**: Регистрация -> Создание профиля -> Переключение на роль Заказчика -> Публикация заказа.
- **Critical Path 2**: Поиск заказа -> Подача отклика (Фрилансер) -> Принятие отклика (Заказчик) -> Сдача работы -> Одобрение и переход средств.
- **Admin Panel**: Управление пользователями, блокировка, корректировка баланса.

## 4. Требования
- Тесты должны запускаться на чистой БД (использовать `db_clear.py` или штатные средства Django/Docker).
- Покрытие кода (coverage) бэкенда не менее 80%.
- Все тесты должны проходить в headless режиме в CI.
- Использование фикстур для генерации тестовых данных.
